# Andromeda — Behavior Audit

Independent verification of the claims in `README.md`, `PAYMYAGENT.md`, and `docs/BUILD-SUMMARY.md`.

Auditor stance: black box. Code was read but not modified. Each test gate was run from a clean port state with `MOCK_MODE=true` (default). HTTP endpoints were probed manually with `curl`. MCP tool registry was diffed against the doc table by reading `mcp/server.js`.

### Provider error-envelope shape (consistency)

Two shapes coexist:

- **Pino-traced routes** return `{ "error": "<code>", "message": "...", "request_id": "req_...", "docs": "..." }`.
- **A few hand-rolled responses** (e.g. `GET /api/v1/subscriptions/:id` for not-found) return only `{"error":"no such subscription"}` — no `request_id`, no `docs`, no `message`. Drift bug. P2.

The exact header name is `x-andromeda-sig` (NOT `x-andromeda-signature`). The registry's diagnostic message ("missing signature headers") would be more useful if it said which header was missing — easy to mis-key with the wrong name.

## 7. Mock-mode verification

Procedure: confirmed `MOCK_MODE=true` in `provider/.env.local` and `buyer/.env`; ran `npm run demo:multi`; inspected stdout, the source code paths that import `@getalby/sdk`, and the `MOCK` branch in each.

- `provider/src/lib/wallet.ts:60-66` — `nwcClient()` is the only constructor call site; gated behind `wallet()` selector at line 88: `process.env.MOCK_MODE === "true" ? mock : real`. Mock branch never touches NWC.
- `buyer/lumen.js:24-30` — `ln()` returns nothing if `MOCK` is set; the `pay()` function at line 34 hits `/api/dev/pay` in mock mode. NWC client is never instantiated.
- `mcp/lumen-client.js:30-36` — same pattern; `ln()` returns null when `MOCK`, never hits `LNClient` constructor.
- Top-level `import { LNClient } from "@getalby/sdk"` does not open a network connection — `LNClient`/`NWCClient` are class exports.

`npm run demo:multi` ran in 1.55 s end-to-end with zero outbound network connections to NWC relays (output shows only `localhost` traffic + the registry-self-register attempt that fails because the registry isn't running for plain `npm run demo:multi`).

**Mock-mode integrity: confirmed.**

## 8. Bugs found, severity-ranked

### P0 · Provider's three subscription sub-routes return Next.js HTML 404

- `POST /api/v1/subscriptions/:id/topup`
- `POST /api/v1/subscriptions/:id/cancel`
- `GET /api/v1/subscriptions/:id/alerts?since=`

Reproduction (from a clean clone with `npm run provider`):

```
$ curl -i -X POST -H 'content-type: application/json' \
       -d '{"sats":50}' \
       http://localhost:3000/api/v1/subscriptions/sub_does_not_matter/topup
HTTP/1.1 404 Not Found
Content-Type: text/html; charset=utf-8
…<html>…<title>404: This page could not be found.</title>…
```

The route files exist:

- `provider/src/app/api/v1/subscriptions/[id]/topup/route.ts` (exports `POST`)
- `provider/src/app/api/v1/subscriptions/[id]/cancel/route.ts` (exports `POST`)
- `provider/src/app/api/v1/subscriptions/[id]/alerts/route.ts` (exports `GET`, has `dynamic = "force-dynamic"`)

The provider's log shows the request *does* reach the dev server (`POST /api/v1/subscriptions/sub_…/topup 404 in 8.6s`), so the route is being recognized — but the response Next.js returns is the framework's not-found HTML, not the JSON body the route function builds via `errorResponse(...)` / `Response.json(...)`. The sibling `GET /api/v1/subscriptions/:id/route.ts` works fine, as does `POST /api/v1/subscribe/route.ts`. Likely a Next.js 16 / Turbopack regression interacting with `dynamic = "force-dynamic"` plus the parent `[id]/route.ts` not exporting POST — but that's a hypothesis, not a fix. Either way, the documented endpoints don't produce documented responses.

Consequence:

- The buyer flow that opens a subscription with the provider's `vision-oracle-3` cannot top up, cancel, or poll alerts via these endpoints.
- BUILD-SUMMARY explicitly lists these in the provider's table.
- No phase test gate exercises these routes on the provider — `test-phase2.js` runs the *market-monitor* agent on port 3100, which is a plain Node http server (different code path) and works fine. The provider routes have no test coverage and are silently broken.

### P1 · `andromeda_purchase_dataset` real-mode is not implemented

`mcp/server.js:660-662`:

```js
} else {
  return fail(`real-mode dataset payment not implemented in MCP yet (NWC route)`);
}
```

The doc table promises "5000 sat (NOAA) · L402-paywalled dataset purchase" with no caveat. BUILD-SUMMARY's "known limitations" section lists 8 items but this isn't one of them. Doc-vs-code drift. (Real-mode also stays unrun by `test-phase6.js`, which is mock-only.)

### P1 · Legacy `scripts/test-phase1.js` fails on a clean clone

Returns `FAIL · 1 of 16 checks failed` because the `/api/v1/stats` admin gate refuses basic auth — `LUMEN_ADMIN_USER` / `LUMEN_ADMIN_PASS` are not configured in `provider/.env.local`. The test sets MOCK_MODE but not admin creds. The script is in the BUILD-SUMMARY table as "(legacy single-provider, intact)" — "intact" is misleading since it actually fails. Either the test should set the env vars before spawning the provider, or the doc should say "fails on clean clone, fix is to set admin creds first". P1 because no Phase-N gate depends on it; P0-adjacent because the "intact" wording is incorrect.

### P2 · MCP tool count discrepancy in BUILD-SUMMARY prose

Body says "7 deprecated `lumen_*` aliases + 23 canonical `andromeda_*` tools = **30 total**." Code registers **24** canonical + **7** deprecated = **31 total**. The table immediately above the prose actually lists 24 canonical names — the "23" is just a typo in the running text.

### P2 · Provider error envelope is inconsistent

Some provider routes return the standardized envelope:

```json
{ "error": "<code>", "message": "<human>", "request_id": "req_xxx", "docs": "..." }
```

…while a few return only `{"error":"no such subscription"}` (e.g. `GET /api/v1/subscriptions/:id`). One service shouldn't ship two envelope shapes.

### P2 · Registry vs. provider error envelopes diverge

Registry uses `{"error":"<human message>"}`. Provider uses `{error, message, request_id, docs}`. Same repo, two sub-services, two envelope contracts. Document one or pick one.

### P2 · Registry signed-request rejection message is generic

When a non-existent header (like the wrong header name `x-andromeda-signature` instead of `x-andromeda-sig`) is sent, the registry returns `missing signature headers` with no indication of which header was missing. Easy to mis-debug.

## 9. Discrepancies between docs and code

1. **README.md "Endpoints" table** (line ~155) lists only 5 provider endpoints (`/api/health`, `/api/v1/discovery`, `/api/v1/listing-verify`, `/api/v1/order-receipt`, `/api/dev/pay`). It does NOT mention `/api/v1/stats`, `/api/v1/receipts/{id}`, `/api/v1/subscribe`, `/api/v1/subscriptions/*`, or `/api/dev/fire-alert`, all of which BUILD-SUMMARY claims and most of which are wired. The README is a stale subset.
2. **PAYMYAGENT.md** still describes "Six tools are exposed to the agent" / "12 checks; should print `PASS · 12/12`" but BUILD-SUMMARY says the MCP now ships 24 canonical + 7 alias tools. PAYMYAGENT.md was not updated when phases 1–6 added 18 more tools. Outdated.
3. **PAYMYAGENT.md** `lumen_status` / `lumen_discover` / etc. are described without the "deprecated alias of …" prefix the code now adds. A user reading PAYMYAGENT.md might think `lumen_*` is canonical; the code says otherwise.
4. **BUILD-SUMMARY.md** "23 canonical = 30 total" — actually 24, total 31 (P2 above).
5. **BUILD-SUMMARY.md** shows test-phase1.js as `(legacy single-provider, intact)`. It is NOT intact — it fails 1 of 16 checks on a clean clone (P1 above).
6. **BUILD-SUMMARY.md** lists the three broken `/api/v1/subscriptions/:id/*` provider routes as if they work. They return the Next.js not-found page (P0 above).
7. **`docs/BUILD-SUMMARY.md` known-limitation list** does not mention that `andromeda_purchase_dataset` returns an error in real mode (P1 above).
8. **README.md** instructs `MAX_PRICE_SATS` cap defaults differ between MCP (4000) and the buyer (4000) and the docs (consistent), but `MAX_BUDGET_SATS` is documented as 5000 in PAYMYAGENT.md and `mcp/budget.js` agrees — minor: just confirming, not a bug.
9. **Provider's `/api/v1/stats` admin gate** is documented as "basic-auth" but on a clean clone it always 401s ("admin disabled (set LUMEN_ADMIN_USER / LUMEN_ADMIN_PASS)"). The docs should state that the admin creds must be configured or the endpoint is closed.
10. **Macaroon `docs` URL** in error envelopes points to `https://github.com/ouazmourad/lumen#errors` — a personal fork URL, not the canonical project URL. Likely a leftover from initial development. Cosmetic.
